1. HOW ARE FLOOR PLANS CURRENTLY STORED?
─────────────────────────────────────────────────────────────
Floor plans are stored in a MongoDB collection called `floor_plans`.

On the backend:
- Collection indexed in backend/app/db/indexes.py (lines 84-87)
- Index is on ("floor_id", "is_active") — one active floor plan per floor
- Uploaded via POST /uploads/floorplans, then saved via POST /floor-plans (workflow.py)
- Files go to Cloudinary: horizon/floorplans/{org_id}/{entity_id}/

On the frontend, the type is MockFloorPlan (frontend/src/data/mockData.ts lines 132-144):
  id, projectId, towerId, floorId, floorLabel, uploadedBy, uploadedAt,
  fileType ('pdf' | 'png' | 'jpg'), fileName, fileSizeMb,
  rooms: MockRoomMarker[]   ← this is the pin/annotation JSON array

IS THERE A JSON FIELD FOR PINS?
Yes. The `rooms` field on MockFloorPlan is an array of MockRoomMarker objects
(mockData.ts lines 146-158):
  id, name, number, type, x, y, width, height, status,
  captureId?, tourId?
These store rectangular marker positions on the floor plan image.

2. HOW ARE ROOM POLYGONS STORED?
─────────────────────────────────────────────────────────────
Room markers/polygons are stored inside the floor plan document as an array
(the `rooms: MockRoomMarker[]` field described above).

Each marker has: x, y, width, height — rectangular regions, not polygons.
They are persisted to the MongoDB floor_plans collection when a floor plan
is saved (workflowStore.ts uploadFloorPlan function, line ~797).

There is NO separate room polygon collection. Polygons are embedded in the
floor plan document.

HOWEVER — the WfRoom objects in the `rooms` collection (workflowStore.ts
lines 48-57) are separate entities that track room metadata (name, type,
flatId, etc.) but do NOT store coordinates. They are linked via `floorPlanId?`.

Conclusion: coordinates live in floor_plans.rooms[]. Room metadata lives in
the rooms collection. The two are linked by captureId / roomId cross-reference.

3. HOW ARE UPLOADS ASSOCIATED?
─────────────────────────────────────────────────────────────
When a floor plan image is uploaded, it is owned by a FLOOR.

Foreign key hierarchy:
  floorId   → directly links to the Floor (primary FK)
  towerId   → denormalised for query performance
  projectId → denormalised for query performance

Evidence:
- FloorPlanUploadPage.tsx reads projectId, towerId, floorId from URL params (line ~61)
- The WfFloor interface has floorPlanId?: string back-reference (workflowStore.ts lines 29-35)
- MongoDB index: ("floor_id", "is_active") — one floor plan per floor

Capture uploads are also owned by a Floor (via roomId → room.floorId).
Upload path: POST /uploads/captures — returns Cloudinary URLs stored on the
Capture document as mediaAssets[].

─────────────────────────────────────────────────────────────

Step 2 — Tour Generation
  Function: generateTour(captureId) — workflowStore.ts line ~730

  NOT a placeholder — creates a real tour object.

Step 3 — Tour Publish
  Function: publishTour(id) — workflowStore.ts line ~758
  Endpoint: PUT /tours/{id}/status (workflow.py line ~459)
  Changes tour status: 'processing' → 'published'

Full review pipeline: uploaded → assigned → reviewing → approved → published
(mockData.ts line ~313)

7. DOES THE APP DETECT DEVICE TYPE?
─────────────────────────────────────────────────────────────
NO explicit device detection exists anywhere in the codebase.

No usage of:

- window.innerWidth breakpoint detection hooks
- isMobile / isTablet / isDesktop variables
- Any device-detection library

Responsive layout IS handled via Material-UI Grid breakpoints:
  xs, sm, md, lg — used across all pages
Example: <Grid size={{ xs: 12, md: 8 }}>

If device-specific behaviour is needed (e.g. camera vs upload),
it must be implemented fresh.

8. IS THERE A CAMERA PAGE USING getUserMedia?
─────────────────────────────────────────────────────────────
NO. There is no camera page and no usage of navigator.mediaDevices.getUserMedia()
anywhere in the codebase.

The app is entirely FILE-UPLOAD-BASED:
- Captures: CaptureUploadPage.tsx (drag-drop or file picker)
- Floor plans: FloorPlanUploadPage.tsx (drag-drop or file picker)
- Avatars: UserProfilePage.tsx (file picker via <input type="file">)

Accepted capture formats: JPG, JPEG, PNG, DNG, INSP, INSV (360° camera exports)
The app expects files exported FROM a physical 360° camera, not live capture.

Camera functionality is NOT implemented and would need to be built from scratch
using navigator.mediaDevices.getUserMedia() if required.

12. CAN THE DATABASE SCHEMA BE MODIFIED?
─────────────────────────────────────────────────────────────
YES, freely. The database is MongoDB (schemaless NoSQL).

There is NO Alembic, NO migrations, NO schema enforcement.

Schema is defined by:
  1. backend/app/db/indexes.py — MongoDB indexes (created at startup, idempotent)
  2. Pydantic models in backend/app/models/ — validation only, not enforced by DB
  3. What the API endpoints write to collections

To add a new field to any collection:
  - Just start writing it — MongoDB accepts any document shape
  - Add an index in indexes.py if querying by that field (safe to re-run)
  - Update the Pydantic model for validation
  - No migration scripts needed, no downtime

To add a new collection:
  - Add indexes in indexes.py
  - Add API endpoints in workflow.py or a new router
  - Add frontend types + store actions

This means Claude CAN design and implement new models/collections freely.
